# 2.4 命名空间与 RBAC

> 本节目标：掌握 Namespace 隔离和 RBAC 权限控制。

---

## 一、Namespace（命名空间）

### 概念

Namespace 是 K8s 中的**逻辑隔离**单元，用于将集群资源划分给不同团队/环境。

### 默认命名空间

| 名称 | 用途 |
|------|------|
| default | 未指定 namespace 时的默认空间 |
| kube-system | K8s 系统组件（API Server、CoreDNS 等） |
| kube-public | 公共资源，所有用户可读 |
| kube-node-lease | 节点心跳租约 |

### 隔离范围

**Namespace 隔离的**：Pod、Service、ConfigMap、Secret、Deployment、PVC 等
**Namespace 不隔离的**：Node、PV、StorageClass、ClusterRole 等集群级资源

### 常用操作

```bash
# 查看命名空间
kubectl get ns

# 创建命名空间
kubectl create ns game-dev
kubectl create ns game-prod

# 在指定命名空间操作
kubectl get pods -n game-dev
kubectl apply -f deploy.yaml -n game-dev

# 设置默认命名空间（避免每次 -n）
kubectl config set-context --current --namespace=game-dev

# 跨命名空间 Service 访问
# 格式：<service-name>.<namespace>.svc.cluster.local
# 例如：mysql-svc.game-prod.svc.cluster.local
```

## 二、RBAC（基于角色的访问控制）

### 核心概念

```
                    ┌──────────────┐
                    │   Subject    │
                    │ (User/SA/Group)
                    └──────┬───────┘
                           │
                    ┌──────┴───────┐
                    │ RoleBinding  │  ← 将角色绑定到主体
                    └──────┬───────┘
                           │
                    ┌──────┴───────┐
                    │    Role      │  ← 定义权限（resources + verbs）
                    └──────────────┘
```

### Role vs ClusterRole

| 类型 | 作用范围 | 用途 |
|------|----------|------|
| Role | 命名空间内 | 控制 ns 内资源权限 |
| ClusterRole | 集群范围 | 控制集群级资源或跨 ns 权限 |
| RoleBinding | 命名空间内 | 绑定 Role/ClusterRole 到主体 |
| ClusterRoleBinding | 集群范围 | 绑定 ClusterRole 到主体 |

### verbs（操作动词）

| 动词 | 含义 |
|------|------|
| get | 获取单个资源 |
| list | 列出资源 |
| watch | 监听变化 |
| create | 创建 |
| update | 更新 |
| patch | 部分更新 |
| delete | 删除 |
| deletecollection | 批量删除 |

## 三、完整实操

### 步骤 1：创建命名空间

```bash
kubectl create ns game-dev
kubectl create ns game-prod

kubectl get ns
# NAME          STATUS   AGE
# default       Active   1d
# game-dev      Active   5s
# game-prod     Active   5s
# kube-system   Active   1d
```

### 步骤 2：创建只读 ServiceAccount 和 Role

```yaml
# rbac-readonly.yaml

# 1. 创建 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: dev-reader
  namespace: game-dev

---
# 2. 创建只读 Role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: readonly-role
  namespace: game-dev
rules:
- apiGroups: [""]                    # 核心 API 组
  resources: ["pods", "services", "configmaps", "secrets"]
  verbs: ["get", "list", "watch"]    # 只允许读操作
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets", "statefulsets"]
  verbs: ["get", "list", "watch"]

---
# 3. 将 Role 绑定到 ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: readonly-binding
  namespace: game-dev
subjects:
- kind: ServiceAccount
  name: dev-reader
  namespace: game-dev
roleRef:
  kind: Role
  name: readonly-role
  apiGroup: rbac.authorization.k8s.io
```

```bash
kubectl apply -f rbac-readonly.yaml
```

### 步骤 3：部署测试 Pod 并验证权限

```bash
# 在 game-dev 部署一个测试 Pod
kubectl run test-pod --image=nginx:1.25 -n game-dev

# 创建一个使用 dev-reader SA 的 Pod 来测试权限
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: rbac-test
  namespace: game-dev
spec:
  serviceAccountName: dev-reader
  containers:
  - name: kubectl
    image: bitnami/kubectl:latest
    command: ["sleep", "3600"]
EOF
```

### 步骤 4：验证权限限制

```bash
# 读操作 — 应该成功 ✅
kubectl exec -n game-dev rbac-test -- kubectl get pods -n game-dev
# NAME        READY   STATUS    RESTARTS   AGE
# test-pod    1/1     Running   0          1m
# rbac-test   1/1     Running   0          30s

# 删除操作 — 应该被拒绝 ❌
kubectl exec -n game-dev rbac-test -- kubectl delete pod test-pod -n game-dev
# Error from server (Forbidden): pods "test-pod" is forbidden:
# User "system:serviceaccount:game-dev:dev-reader" cannot delete
# resource "pods" in API group "" in the namespace "game-dev"  ✅

# 验证也无法创建
kubectl exec -n game-dev rbac-test -- kubectl run new-pod --image=nginx -n game-dev
# Error from server (Forbidden)...  ✅
```

### 快捷验证方式：kubectl auth can-i

```bash
# 检查 dev-reader 是否能执行特定操作
kubectl auth can-i get pods \
  --as=system:serviceaccount:game-dev:dev-reader \
  -n game-dev
# yes  ✅

kubectl auth can-i delete pods \
  --as=system:serviceaccount:game-dev:dev-reader \
  -n game-dev
# no  ✅

kubectl auth can-i create deployments \
  --as=system:serviceaccount:game-dev:dev-reader \
  -n game-dev
# no  ✅
```
