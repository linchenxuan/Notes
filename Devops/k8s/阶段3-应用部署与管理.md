## 🏗️ StatefulSet有状态应用：数据库集群的守护者

### 📖 技术背景

StatefulSet专门为有状态应用设计，解决了传统Deployment在部署数据库集群时面临的挑战：数据持久性、网络标识稳定性、有序部署等问题。

### 🎯 核心概念

StatefulSet专门为有状态应用设计，提供了以下关键特性：
- **稳定网络标识**：每个Pod都有固定的主机名和网络标识
- **持久化存储**：Pod重启后仍能访问相同的数据
- **有序部署和扩缩容**：按照顺序创建和删除Pod
- **有序更新**：可控的滚动更新策略

### 💡 技术类比

Deployment类似于无状态服务池，而StatefulSet则提供稳定的网络标识和存储绑定，适合需要持久化状态的应用场景。

## 🔧 DaemonSet节点级服务：集群守护进程

### 📖 技术应用场景

DaemonSet确保在每个节点上都运行一个Pod副本，适用于需要在集群每个节点部署代理服务的场景，如日志收集、监控代理等基础设施组件。

### 🎯 典型用途

- **日志收集**：Fluentd、Filebeat、Logstash
- **监控代理**：Prometheus Node Exporter、Datadog Agent
- **网络插件**：Calico、Flannel、Weave Net
- **安全工具**：Falco、Aqua Security

## ⏰ Job和CronJob：批处理任务调度

### 📖 批处理应用场景

CronJob适用于定时任务场景，如数据库备份、数据清理、批量处理等周期性操作，支持任务成功通知、失败重试等可靠性机制。

### 🎯 使用场景

- **定时备份**：数据库、文件系统备份
- **数据清理**：清理过期日志、临时文件
- **批量处理**：图片处理、数据转换

## 💡 生产实践总结

### 🎯 工作负载选择指南

基于技术特性和应用场景，工作负载选择应遵循以下决策原则：

**1. 无状态应用 ➜ Deployment**
- Web应用、API服务、微服务
- 可以水平扩展，无数据依赖

**2. 有状态应用 ➜ StatefulSet**
- 数据库、消息队列、缓存集群
- 需要稳定网络标识和持久化存储

**3. 节点级服务 ➜ DaemonSet**
- 日志收集、监控代理、网络插件
- 需要在每个节点上运行

**4. 批处理任务 ➜ Job/CronJob**
- 数据处理、定时任务、备份作业
- 一次性或周期性执行

## 🚀 Deployment高级特性：从基础到精通

### 📖 技术背景

Deployment作为Kubernetes中最常用的工作负载，提供了强大的部署管理功能。深入理解其高级特性对于构建可靠的云原生应用至关重要。

### 🎯 核心概念

Deployment不仅仅是简单的副本管理器，它还提供了：
- **滚动更新**：零停机时间更新应用
- **版本回滚**：快速回退到稳定版本
- **扩缩容**：根据负载自动调整副本数
- **健康检查**：确保应用稳定运行

#### Deployment核心概念回顾
```yaml
# Deployment配置示例
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp-deployment
  labels:
    app: webapp
    version: v1.0
spec:
  # 1. 副本管理
  replicas: 3
  
  # 2. 选择器 - 必须匹配Pod标签
  selector:
    matchLabels:
      app: webapp
  
  # 3. 更新策略（重要！）
  strategy:
    type: RollingUpdate  # 或 Recreate
    rollingUpdate:
      maxUnavailable: 25%  # 更新时最大不可用比例
      maxSurge: 25%        # 更新时最大激增比例
  
  # 4. 修订历史限制
  revisionHistoryLimit: 10  # 保留的历史版本数
  
  # 5. 进度截止时间
  progressDeadlineSeconds: 600  # 更新超时时间
  
  template:
    metadata:
      labels:
        app: webapp  # 必须匹配selector
    spec:
      containers:
      - name: webapp
        image: nginx:1.23
        ports:
        - containerPort: 80
```

#### 滚动更新实践
```bash
# 滚动更新操作示例

# 1. 查看当前Deployment状态
kubectl get deployment webapp-deployment
# 输出：READY 3/3，表示3个副本都正常运行

# 2. 更新镜像版本
kubectl set image deployment/webapp-deployment webapp=nginx:1.24

# 3. 查看更新状态
kubectl rollout status deployment/webapp-deployment
# 实时显示更新进度

# 4. 查看更新历史
kubectl rollout history deployment/webapp-deployment
# 显示：
# deployment.apps/webapp-deployment
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         kubectl set image deployment/webapp-deployment webapp=nginx:1.24

# 5. 回滚到上一个版本
kubectl rollout undo deployment/webapp-deployment

# 6. 回滚到指定版本
kubectl rollout undo deployment/webapp-deployment --to-revision=1
```

**技术要点总结**：
- RollingUpdate：逐个替换Pod，保证服务不中断
- Recreate：先删除所有旧Pod，再创建新Pod，会有短暂中断
- maxUnavailable：控制更新时的不可用Pod数量
- maxSurge：控制更新时可以额外创建的Pod数量

### 💡 更新策略配置建议

不同业务场景需要配置相应的更新策略：

**电商大促期间**：
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0%   # 不允许任何不可用
    maxSurge: 100%       # 先创建新Pod，再删除旧Pod
```

**内部管理系统**：
```yaml
strategy:
  type: Recreate  # 允许短暂中断，快速更新
```

**高并发服务**：
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 10%  # 保守更新
    maxSurge: 10%        # 控制资源使用
```

#### 更新策略对比实验
```yaml
# 实验1：快速更新（激进策略）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fast-update
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 50%  # 允许50%不可用
      maxSurge: 100%       # 允许100%激增
  replicas: 4

---
# 实验2：保守更新（安全策略）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: safe-update
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 10%  # 只允许10%不可用
      maxSurge: 10%        # 只允许10%激增
  replicas: 10
```

### 📈 HPA自动扩缩容：智能弹性伸缩的奥秘

#### 🎯 核心概念

HPA（Horizontal Pod Autoscaler）是Kubernetes中最实用的功能之一，它能够根据CPU、内存等指标自动调整Pod副本数。我第一次配置HPA时，看到Pod数量根据负载自动增减，感觉就像在观看魔术表演一样神奇。

#### 💡 技术原理

HPA的工作机制：
1. **指标收集**：通过metrics-server收集Pod资源使用数据
2. **算法计算**：根据当前指标和目标值计算期望副本数
3. **扩缩容执行**：调用Deployment的Scale接口调整副本数
4. **稳定窗口**：避免频繁扩缩容造成的抖动

#### 🚀 实战案例：电商秒杀系统

HPA在电商秒杀系统中能够实现智能弹性伸缩，应对突发流量：

#### HPA配置示例
```yaml
# HPA配置示例
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webapp-hpa
spec:
  # 1. 目标Deployment
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp-deployment
  
  # 2. 最小和最大副本数
  minReplicas: 2
  maxReplicas: 10
  
  # 3. 指标配置
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # CPU使用率70%时扩容
  
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80  # 内存使用率80%时扩容
  
  # 4. 扩缩容行为配置
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60  # 扩容稳定窗口
      policies:
      - type: Percent
        value: 100  # 每次扩容100%
        periodSeconds: 15
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容稳定窗口（5分钟）
      policies:
      - type: Percent
        value: 10  # 每次缩容10%
        periodSeconds: 60
```

#### HPA实践操作
```bash
# HPA配置和操作示例

# 1. 安装metrics-server（需要先安装）
minikube addons enable metrics-server

# 2. 创建测试Deployment
kubectl apply -f webapp-deployment.yaml

# 3. 创建HPA
kubectl apply -f webapp-hpa.yaml

# 4. 查看HPA状态
kubectl get hpa
# 输出：
# NAME         REFERENCE                       TARGETS         MINPODS   MAXPODS   REPLICAS   AGE
# webapp-hpa   Deployment/webapp-deployment   <unknown>/70%   2         10        0          10s

# 5. 等待指标收集（约1-2分钟）
# 再次查看：
# webapp-hpa   Deployment/webapp-deployment   0%/70%          2         10        2          2m

# 6. 生成负载测试
kubectl run load-test --image=busybox --rm -it -- /bin/sh -c "while true; do wget -q -O- http://webapp-service; done"

#### 💡 常见问题分析：HPA配置技术要点

**问题1：HPA不生效**
```bash
# 错误：HPA显示<unknown>/70%
kubectl get hpa
```
**技术原理**：metrics-server未安装或配置错误
**解决方法**：
```bash
# 检查metrics-server状态
kubectl get pods -n kube-system | grep metrics-server

# 重新安装metrics-server
minikube addons enable metrics-server
```

**问题2：扩缩容过于频繁**
```yaml
# 错误配置：稳定窗口时间太短
behavior:
  scaleUp:
    stabilizationWindowSeconds: 10  # 太短了！
```
**解决方案**：
```yaml
# 正确的稳定窗口配置
behavior:
  scaleUp:
    stabilizationWindowSeconds: 60  # 至少60秒
  scaleDown:
    stabilizationWindowSeconds: 300  # 缩容要更加保守
```

**问题3：扩容后服务仍然不可用**
**技术原理**：只配置了HPA，但没有配置Pod的resource requests
**解决方法**：
```yaml
# Deployment中必须配置resource requests
spec:
  template:
    spec:
      containers:
      - name: webapp
        resources:
          requests:
            cpu: 100m    # 必须配置！
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 512Mi
```

# 7. 观察扩容过程
kubectl get hpa -w
# 可以看到副本数从2增加到4、6、8、10

# 8. 停止负载后观察缩容
# 副本数会逐渐减少到最小值2
```

**技术要点**：
- HPA根据CPU/内存使用率自动调整副本数
- 扩容反应快（1分钟内），缩容反应慢（5分钟后）
- 需要安装metrics-server才能获取指标
- 可以配置多个指标和复杂的扩缩容策略

## Week 7：StatefulSet有状态应用

### 🗃️ Day 7-8：StatefulSet基础概念

#### StatefulSet vs Deployment对比
```yaml
# Deployment（无状态）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-deployment
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: web
        image: nginx:1.23

---
# StatefulSet（有状态）
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web-statefulset
spec:
  serviceName: "web-service"  # 必须指定headless service
  replicas: 3
  template:
    spec:
      containers:
      - name: web
        image: nginx:1.23
  
  # StatefulSet特有配置
  volumeClaimTemplates:  # 存储模板
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 1Gi
```

#### StatefulSet特性学习
```bash
# StatefulSet操作示例

# 1. 创建StatefulSet
kubectl apply -f web-statefulset.yaml

# 2. 观察Pod创建顺序（有序创建）
kubectl get pods -l app=web -w
# 可以看到Pod按顺序创建：web-0, web-1, web-2

# 3. 查看Pod名称和网络标识
kubectl get pods -l app=web -o wide
# Pod名称：web-statefulset-0, web-statefulset-1, web-statefulset-2

# 4. 测试DNS解析（需要创建headless service）
kubectl exec -it web-statefulset-0 -- nslookup web-statefulset-1.web-service
# 可以解析到其他Pod的DNS名称

# 5. 测试存储持久性
kubectl exec -it web-statefulset-0 -- sh -c 'echo "test data" > /usr/share/nginx/html/test.txt'
kubectl delete pod web-statefulset-0
# 等待新Pod创建
kubectl exec -it web-statefulset-0 -- cat /usr/share/nginx/html/test.txt
# 数据仍然存在！

# 6. 观察Pod删除顺序（有序删除）
kubectl delete statefulset web-statefulset
# Pod按逆序删除：web-2, web-1, web-0
```

**技术要点总结**：
- **稳定网络标识**：Pod有固定的DNS名称和序号
- **有序部署/删除**：按序号顺序创建和删除Pod
- **持久存储**：每个Pod有独立的持久化存储
- **适用于**：数据库、分布式存储等有状态应用

#### 完整的StatefulSet配置
```yaml
# StatefulSet完整配置示例
apiVersion: v1
kind: Service
metadata:
  name: web-headless
  labels:
    app: web
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None  # Headless service
  selector:
    app: web

---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web-statefulset
spec:
  serviceName: "web-headless"
  replicas: 3
  selector:
    matchLabels:

#### CronJob定时任务
```yaml
# 我的CronJob学习笔记
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup-cronjob
spec:
  # 1. 调度表达式（cron格式）
  schedule: "0 2 * * *"  # 每天凌晨2点执行
  
  # 2. 时区配置
  timeZone: "Asia/Shanghai"
  
  # 3. 并发策略
  concurrencyPolicy: Forbid  # Forbid, Allow, Replace
  
  # 4. 历史限制
  successfulJobsHistoryLimit: 3    # 保留成功Job的历史记录数
  failedJobsHistoryLimit: 1        # 保留失败Job的历史记录数
  
  # 5. 暂停配置
  suspend: false  # 是否暂停执行
  
  # 6. 执行截止时间
  startingDeadlineSeconds: 300  # 如果错过调度时间，300秒内仍可执行
  
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: backup
            image: mysql:8.0
            command:
            - /bin/sh
            - -c
            - |
              echo "Starting backup at $(date)"
              mysqldump -h mysql-service -u root -p$MYSQL_ROOT_PASSWORD mydb > /backup/backup-$(date +%Y%m%d-%H%M%S).sql
              echo "Backup completed at $(date)"
            env:
            - name: MYSQL_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: mysql-root-password
            volumeMounts:
            - name: backup-storage
              mountPath: /backup
          volumes:
          - name: backup-storage
            persistentVolumeClaim:
              claimName: backup-pvc
```
