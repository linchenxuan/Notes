# 3.4 资源管理与调度

> 本节目标：掌握资源限制、QoS、HPA 自动扩缩容和调度策略。

---

## 一、Requests 和 Limits

### 概念

| 参数 | 含义 | 调度 | 运行时 |
|------|------|------|--------|
| requests | Pod 运行**最低保障**资源 | Scheduler 依据此调度 | 保证可用 |
| limits | Pod 能使用的**最大**资源 | 不参与调度 | 超出则限制/OOMKill |

### CPU 单位

- `1` = 1 个 vCPU 核心
- `500m` = 0.5 核心（m = 千分之一核）
- `100m` = 0.1 核心

### Memory 单位

- `128Mi` = 128 MiB
- `1Gi` = 1 GiB
- `256M` = 256 MB（注意 Mi 和 M 的区别）

### 设置建议

```yaml
resources:
  requests:
    cpu: "100m"      # 基准：观察平时用量
    memory: "128Mi"  # 基准：启动后稳定用量
  limits:
    cpu: "500m"      # 峰值：requests 的 2-5 倍
    memory: "256Mi"  # 峰值：requests 的 1.5-2 倍（OOM 更危险）
```

> **建议**：Memory limits 不要设太高于 requests，否则 OOM 时影响大。CPU 可以适当放宽。

## 二、QoS 等级

K8s 根据 requests/limits 设置自动为 Pod 分配 QoS 等级，影响 OOM 驱逐优先级。

| 等级 | 条件 | OOM 驱逐顺序 |
|------|------|-------------|
| **Guaranteed** | 所有容器的 CPU 和 Memory 都设置了 requests = limits | 最后被驱逐 |
| **Burstable** | 至少一个容器设置了 requests 或 limits，但不满足 Guaranteed | 中间 |
| **BestEffort** | 没有设置任何 requests 和 limits | 最先被驱逐 |

```yaml
# Guaranteed（推荐用于关键游戏服务）
resources:
  requests:
    cpu: "500m"
    memory: "256Mi"
  limits:
    cpu: "500m"       # = requests
    memory: "256Mi"   # = requests

# Burstable（常规服务）
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"       # > requests
    memory: "256Mi"

# BestEffort（不推荐，不设置任何 resources）
```

## 三、LimitRange 和 ResourceQuota

### LimitRange（命名空间级别默认值和限制）

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: game-limits
  namespace: game-dev
spec:
  limits:
  - type: Container
    default:           # 默认 limits
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:    # 默认 requests
      cpu: "100m"
      memory: "128Mi"
    max:               # 最大值
      cpu: "2"
      memory: "1Gi"
    min:               # 最小值
      cpu: "50m"
      memory: "64Mi"
```

### ResourceQuota（命名空间资源总量限制）

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: game-quota
  namespace: game-dev
spec:
  hard:
    pods: "50"
    requests.cpu: "10"
    requests.memory: "20Gi"
    limits.cpu: "20"
    limits.memory: "40Gi"
    persistentvolumeclaims: "10"
```

## 四、HPA（Horizontal Pod Autoscaler）

### 工作原理

```
Metrics Server → HPA Controller → 对比 指标 vs 阈值 → 调整 replicas
                                  (每 15s 检查一次)
```

### 前提

```bash
# 安装 Metrics Server（minikube）
minikube addons enable metrics-server

# 验证
kubectl top nodes
kubectl top pods
```

### 支持的指标类型

| 类型 | 示例 |
|------|------|
| Resource | CPU、Memory 使用率 |
| Custom | 自定义 Prometheus 指标 |
| External | 外部指标（如队列长度） |

### 实操：为游戏服务配置 HPA

```yaml
# game-with-hpa.yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: game-gate
spec:
  replicas: 2
  selector:
    matchLabels:
      app: game-gate
  template:
    metadata:
      labels:
        app: game-gate
    spec:
      containers:
      - name: gate
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: "100m"         # HPA 必须设置 requests
            memory: "128Mi"
          limits:
            cpu: "500m"
            memory: "256Mi"
---
apiVersion: v1
kind: Service
metadata:
  name: game-gate-svc
spec:
  selector:
    app: game-gate
  ports:
  - port: 80
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: game-gate-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: game-gate
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60    # CPU > 60% 时扩容
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30   # 扩容等待 30s
      policies:
      - type: Pods
        value: 2                        # 每次最多加 2 个 Pod
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300   # 缩容等待 5 分钟
      policies:
      - type: Pods
        value: 1
        periodSeconds: 60
```

```bash
kubectl apply -f game-with-hpa.yaml

# 查看 HPA
kubectl get hpa
# NAME            REFERENCE              TARGETS   MINPODS   MAXPODS   REPLICAS
# game-gate-hpa   Deployment/game-gate   5%/60%    2         10        2
```

### 压测触发扩容

```bash
# 安装压测工具
# vegeta: go install github.com/tsenart/vegeta@latest
# 或 wrk: apt-get install wrk

# 方式 1：用 kubectl 内置生成负载
kubectl run load-generator --rm -it --image=busybox:1.36 -- /bin/sh
# 在容器内执行：
while true; do wget -q -O- http://game-gate-svc > /dev/null; done

# 方式 2：用 vegeta（从可访问集群的机器）
echo "GET http://localhost:8080" | vegeta attack -rate=500/s -duration=120s | vegeta report

# 方式 3：用 wrk
wrk -t4 -c200 -d120s http://localhost:8080

# 在另一个终端监控 HPA
kubectl get hpa -w
# NAME            TARGETS    REPLICAS
# game-gate-hpa   5%/60%     2
# game-gate-hpa   72%/60%    2        ← CPU 超过阈值
# game-gate-hpa   72%/60%    4        ← 开始扩容
# game-gate-hpa   85%/60%    6        ← 继续扩容
# ...

# 停止压测后等待 5 分钟（stabilizationWindow）
# game-gate-hpa   15%/60%    6
# game-gate-hpa   15%/60%    5        ← 开始缩容
# game-gate-hpa   8%/60%     2        ← 缩回最小副本  ✅
```

## 五、Node Affinity / Taint / Toleration

### Node Affinity（节点亲和性）

```yaml
spec:
  affinity:
    nodeAffinity:
      # 硬约束：必须满足
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-type
            operator: In
            values: ["game-server"]
      # 软约束：优先满足
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values: ["zone-a"]
```

### Taint 和 Toleration

Taint（节点驱离）：阻止 Pod 调度到节点，除非 Pod 有对应 Toleration。

```bash
# 给节点打 Taint
kubectl taint nodes node-1 dedicated=game:NoSchedule

# 查看
kubectl describe node node-1 | grep Taint
# Taints: dedicated=game:NoSchedule
```

```yaml
# Pod 添加 Toleration
spec:
  tolerations:
  - key: "dedicated"
    operator: "Equal"
    value: "game"
    effect: "NoSchedule"
```

| Effect | 含义 |
|--------|------|
| NoSchedule | 新 Pod 不调度到此节点 |
| PreferNoSchedule | 尽量不调度 |
| NoExecute | 新 Pod 不调度 + 已有不匹配 Pod 被驱逐 |

### Pod Anti-Affinity（高可用）

确保同一 Deployment 的 Pod 分散到不同节点：

```yaml
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: game-gate
        topologyKey: kubernetes.io/hostname
```
