# 1.2 容器网络与存储

> 本节目标：理解 Docker 网络模式和存储机制，掌握自定义网络和数据持久化。

---

## 一、Docker 网络模式

### 1. bridge 模式（默认）

每个容器连接到 `docker0` 虚拟网桥，获得独立 IP，通过 NAT 访问外部。

```
┌──────────────────────────────────────────────┐
│                    Host                       │
│                                              │
│   ┌──────────┐  ┌──────────┐                │
│   │Container1│  │Container2│                │
│   │172.17.0.2│  │172.17.0.3│                │
│   └────┬─────┘  └────┬─────┘                │
│        │              │                      │
│   ┌────┴──────────────┴─────┐                │
│   │     docker0 (bridge)     │                │
│   │       172.17.0.1         │                │
│   └────────────┬─────────────┘                │
│                │ NAT                          │
│   ┌────────────┴─────────────┐                │
│   │      eth0 (物理网卡)      │                │
│   └──────────────────────────┘                │
└──────────────────────────────────────────────┘
```

```bash
# 默认使用 bridge
docker run -d --name web nginx

# 查看容器网络
docker inspect web --format '{{.NetworkSettings.IPAddress}}'
# 172.17.0.2
```

### 2. host 模式

容器直接使用宿主机网络栈，无网络隔离，性能最好。

```bash
docker run -d --network host --name web nginx
# 直接通过 localhost:80 访问，无需端口映射
```

**适用场景**：对网络性能要求极高的服务（如游戏服务器的 UDP 通信）。

### 3. none 模式

容器无网络连接，完全隔离。

```bash
docker run -d --network none --name isolated alpine sleep 3600
# 容器内只有 loopback 接口
```

**适用场景**：安全敏感的计算任务、离线数据处理。

### 对比总结

| 模式 | 隔离性 | 性能 | 使用场景 |
|------|--------|------|----------|
| bridge | 高 | 一般（NAT 开销） | 大部分场景（默认） |
| host | 无 | 最好 | 高性能网络需求 |
| none | 完全 | N/A | 安全隔离场景 |

## 二、端口映射

```bash
# 格式：-p 宿主机端口:容器端口
docker run -d -p 8080:80 nginx

# 映射多个端口
docker run -d -p 8080:80 -p 8443:443 nginx

# 映射到指定 IP
docker run -d -p 127.0.0.1:8080:80 nginx

# 随机端口映射
docker run -d -P nginx   # 大写 P，随机映射 EXPOSE 声明的端口
docker port <container>  # 查看映射结果
```

**原理**：Docker 通过 iptables DNAT 规则实现端口映射。

## 三、自定义网络

### 为什么需要自定义网络？

默认 bridge 网络中，容器**不能通过容器名互访**，只能用 IP。自定义网络支持**自动 DNS 解析**。

### 实操步骤

```bash
# 1. 创建自定义网络
docker network create game-net

# 2. 在自定义网络中启动容器
docker run -d --name redis --network game-net redis:7-alpine
docker run -d --name game-server --network game-net alpine sleep 3600

# 3. 通过容器名互相访问
docker exec game-server ping -c 3 redis
# PING redis (172.18.0.2): 56 data bytes
# 64 bytes from 172.18.0.2: seq=0 ttl=64 time=0.098 ms  ✅

# 4. 查看网络详情
docker network inspect game-net

# 5. 清理
docker stop redis game-server
docker rm redis game-server
docker network rm game-net
```

### Bridge 网络下容器间通信数据流

```
Container A (172.18.0.2)                    Container B (172.18.0.3)
      │                                           ▲
      │ 1. 发送数据包                               │ 5. 接收数据包
      ▼                                           │
┌─────────────┐                             ┌─────────────┐
│   veth-A    │                             │   veth-B    │
│ (容器内eth0) │                             │ (容器内eth0) │
└──────┬──────┘                             └──────┬──────┘
       │ 2. 通过 veth pair                          │
       │    到达网桥                                 │ 4. 从网桥转发
       ▼                                           │    到目标 veth
┌──────────────────────────────────────────────────┐
│              docker0 / br-xxxx (网桥)              │
│                                                    │
│  3. 网桥查 MAC 表，转发到目标端口                      │
└──────────────────────────────────────────────────┘

数据流路径：
App A → veth-A → 网桥(MAC 转发) → veth-B → App B
```

## 四、存储：Volume vs Bind Mount

### 对比

| 特性 | Volume | Bind Mount |
|------|--------|------------|
| 存储位置 | Docker 管理（/var/lib/docker/volumes/） | 宿主机任意路径 |
| 创建方式 | `docker volume create` 或自动创建 | 指定宿主机绝对路径 |
| 可移植性 | 高（Docker 管理生命周期） | 低（依赖宿主机路径） |
| 性能 | 高 | 取决于文件系统 |
| **推荐场景** | **数据持久化（数据库等）** | **开发时映射源码** |

### Volume 操作

```bash
# 创建 volume
docker volume create mysql-data

# 查看 volume
docker volume ls
docker volume inspect mysql-data

# 使用 volume
docker run -d --name mysql \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=password \
  mysql:8.0

# 删除 volume
docker volume rm mysql-data

# 清理未使用的 volume
docker volume prune
```

### Bind Mount 操作

```bash
# 将当前目录映射到容器中（开发常用）
docker run -d --name dev-server \
  -v $(pwd)/src:/app/src \
  -p 8080:8080 \
  node:20-alpine npm start
```

## 五、实操：Volume 持久化 MySQL 数据

### 完整步骤

```bash
# 1. 创建 volume
docker volume create game-mysql-data

# 2. 启动 MySQL 容器
docker run -d --name game-mysql \
  -v game-mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=gamepass \
  -e MYSQL_DATABASE=gamedb \
  -p 3306:3306 \
  mysql:8.0

# 3. 等待 MySQL 就绪
docker exec game-mysql mysqladmin ping -h localhost --wait=30 -pgamepass
# mysqld is alive

# 4. 写入测试数据
docker exec -i game-mysql mysql -pgamepass gamedb <<'EOF'
CREATE TABLE players (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    level INT DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
INSERT INTO players (name, level) VALUES ('Player1', 30), ('Player2', 45);
EOF

# 5. 验证数据写入
docker exec game-mysql mysql -pgamepass gamedb -e "SELECT * FROM players;"
# +----+---------+-------+---------------------+
# | id | name    | level | created_at          |
# +----+---------+-------+---------------------+
# |  1 | Player1 |    30 | 2024-01-01 12:00:00 |
# |  2 | Player2 |    45 | 2024-01-01 12:00:00 |
# +----+---------+-------+---------------------+

# 6. 删除容器（模拟故障）
docker stop game-mysql
docker rm game-mysql

# 7. 用同一个 volume 重建容器
docker run -d --name game-mysql \
  -v game-mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=gamepass \
  -p 3306:3306 \
  mysql:8.0

# 8. 验证数据完整保留
docker exec game-mysql mysql -pgamepass gamedb -e "SELECT * FROM players;"
# +----+---------+-------+---------------------+
# | id | name    | level | created_at          |
# +----+---------+-------+---------------------+
# |  1 | Player1 |    30 | 2024-01-01 12:00:00 |
# |  2 | Player2 |    45 | 2024-01-01 12:00:00 |
# +----+---------+-------+---------------------+
# ✅ 数据完整保留！

# 9. 清理
docker stop game-mysql && docker rm game-mysql
docker volume rm game-mysql-data
```
